# RFC: Образ Rule и Skill

**Статус:** Draft v4
**Дата:** 2026-04-17

---

## Мотивация

Агент работает в условиях ограниченного контекстного окна. Каждый токен, загруженный в контекст — это цена. Rules загружаются **всегда**, skills — **по необходимости**. Нужны чёткие критерии, чтобы:

1. Не тратить контекст на то, что сейчас не нужно
2. Не терять критичные знания, которые нужны всегда
3. Новые авторы знали, куда класть новую информацию

---

## Принцип разделения

> **Литмус-тест:** «Хочу ли я, чтобы эта инструкция применялась, даже если я об этом не думаю?»  
> ✅ Да → **Rule**  
> ❌ Нет → **Skill**

---

## Образ Rule

### Что такое Rule

Rule — это **конституция** агента. Набор коротких, безусловных требований, которые агент обязан соблюдать в **каждом** ответе, независимо от задачи.

### Признаки знания, которое должно быть Rule

| Признак | Пример |
|---------|--------|
| Нарушение приводит к **опасным последствиям** | Запрет на `grep -r` из корня Arcadia |
| Это **конвенция/code style**, применимая к любому коду | C++ naming, SQL BIGSERIAL, OpenAPI nullable |
| Это **базовая команда**, используемая в >50% задач | `ya make`, `ya style`, `ya project update` |
| Это **навигация**, без которой агент не может начать работу | Как найти сервис, как запустить сессию |
| Это **мета-инструкция** про работу с другими видами наших кастомизацией| Когда загружать skill документации, когда не загружать |

### Анти-признаки (НЕ должно быть Rule)

| Анти-признак | Пример |
|-------------|--------|
| Пошаговая процедура для конкретной задачи | «Как добавить PostgreSQL БД» |
| Нужно только при работе с конкретной технологией | gRPC streaming, STQ workers |
| Содержит длинные примеры кода (>20 строк) | Пример pg cache header |
| Применяется в <20% задач | Настройка nginx, experiments3 |

### Требования к формату Rule

#### Размер
- **Жёсткий лимит:** один rule-файл ≤ **200 строк**
- **Целевой размер:** 50–100 строк
- **Суммарно все rules:** ≤ **3000 токенов** (~1500 строк)

#### Frontmatter
Отсутствует


#### Содержание
- **Императивный стиль:** «You MUST», «You MUST NOT», «NEVER», «ALWAYS»
- **Без длинных примеров:** максимум 5–10 строк кода на пример
- **Без пошаговых процедур:** если есть шаги 1-2-3-4-5, это skill
- **Без туториалов:** rule — это шпаргалка, не учебник
- **Без прямых ссылок на конкретные skills:** см. ниже «Связь rule и skill»

#### Связь rule и skill

Rule фиксирует **принцип** («что делать всегда»), skill — **процедуру** («как делать, когда возник сценарий»). Прямые ссылки из rule на конкретный skill — антипаттерн:

- Дублируют триггеры, которые уже живут в skill description
- Делают rule хрупким к переименованию/рефакторингу skills
- Тратят always-on токены на подсказку, которая нужна редко

**Правильный способ связать rule и skill — семантически, через description скилла.** Rule говорит, что делать; description скилла ловит момент, когда нужна детальная процедура.

**Пример связки:**

- Rule:
  ```markdown
  # OpenAPI schema changes
  After changing an OpenAPI schema you MUST run codegen
  ```
- Skill `c35-openapi`, description:
  ```yaml
  description: >
    Troubleshooting and procedures for OpenAPI codegen in uservices.
    Load this skill when: you changed an OpenAPI schema and need to run codegen,
    codegen failed, generated files look wrong, or you need to set up codegen
  ```

Rule напоминает о принципе всегда. Skill подхватывается агентом только когда сценарий реально возник (например, пользователь пожаловался на ошибку codegen). Никаких перекрёстных ссылок, никакого дублирования.

**Если агент не загружает нужный skill в подходящий момент** — это проблема description скилла (слабые триггеры), а не повод вставлять ссылку в rule.

#### Структура файла
```markdown
# <Тема> **MANDATORY**

<1-3 предложения: суть правила>

<Короткий пример если нужен>

# <Ещё тема>

...
```

#### Пример хорошего rule
```markdown
# Function complexity
Try to limit function size to 5–20 lines. Decompose bigger functions.

# Keep statements simple
Prefer multiple small statements to one big complex statement.

# Do not return std::pair
`std::pair` says nothing about semantics. Define a struct and return it instead.

# Boolean function naming
A function returning `bool` MUST have a name starting with:
`Is`, `Has`, `Did`, `Can`, `Should`, `Will`, or a verb in 3rd person (`Contains`, `Exists`).
```

~12 строк, ~80 токенов. Агент знает C++ code style всегда — применяется к каждому файлу.

## Образ Skill

### Что такое Skill

Skill — это **экспертиза по запросу**. Пошаговая процедура для конкретной задачи, которая загружается только когда агент видит релевантный запрос пользователя. 

### Признаки знания, которое должно быть Skill

| Признак | Пример |
|---------|--------|
| **Пошаговая процедура** с конкретными действиями | «Как добавить HTTP handler: 1) создать схему, 2) запустить codegen...» |
| Нужно только при работе с **конкретной технологией** | PostgreSQL setup, gRPC auth, STQ workers |
| Содержит **развёрнутые примеры** кода | Пример pg cache policy struct |
| Применяется в **<20% задач** | Настройка nginx, experiments3 |
| Содержит **troubleshooting** | «Если ошибка SSL — включить auth-type: ssl» |

### Требования к Description (frontmatter)

Description — это **единственное**, что агент видит до загрузки skill. От его качества зависит, будет ли skill загружен в нужный момент.

#### Формат
```yaml
---
description: >
  <Что делает — одно предложение>.
  <Включает: перечень тем через запятую>.
  <Триггер: ключевые слова из реальных задач пользователей>.
name: <уникальное имя skill>
---
```

#### Требования
- **Длина:** 2–5 предложений, 30–80 слов
- **Первое предложение:** что skill делает (глагол в инфинитиве)
- **Включает:** перечень конкретных тем, покрытых skill
- **Триггер:** слова и фразы, которые пользователь реально использует в запросах
- **Язык:** английский (для совместимости с routing-моделями)

#### Пример хорошего description
```yaml
description: >
  Step-by-step guide to add PostgreSQL database to a uservice.
  Includes: codegen-module.yaml setup, migrations, pg cache, SQL queries,
  composite types, extensions, postgis.
  Trigger: add database, postgresql, postgres, migration, pg cache,
  sql query, pgcrypto, postgis.
```

#### Пример плохого description
```yaml
# Слишком общий — агент не поймёт когда загружать
description: how to use PostgreSQL database client

# Слишком длинный — тратит токены на discovery
description: >
  This skill provides comprehensive documentation on how to configure,
  set up, and use PostgreSQL databases in uservices including but not
  limited to creating databases, writing migrations, setting up caches,
  writing SQL queries, using composite types, configuring extensions
  like pgcrypto and postgis, and troubleshooting common issues...
```

### Требования к телу Skill (SKILL.md)

#### Размер
- **Рекомендуемый:** 50–300 строк
- **Максимальный:** 500 строк для основного файла
- Если skill больше 500 строк — разбить на под-файлы (см. ниже)

#### Структура
```markdown
---
description: <...>
name: <...>
---

# <Название процедуры> **MANDATORY**

<Пошаговая инструкция с нумерованными шагами>

# <Ещё процедура>

<...>

# Примеры

<Развёрнутые примеры кода>
```

#### Содержание
- **Процедурный стиль:** нумерованные шаги, конкретные команды, конкретные пути
- **Примеры кода:** развёрнутые, с контекстом (в отличие от rules)
- **Troubleshooting:** типичные ошибки и их решения
- **Связь с другими skills — семантически, не ссылками:** если нужно «подсказать» агенту соседний skill в сценарии — повторяем ключевые формулировки этого сценария в теле текущего skill **и** в description того, на который хотим направить. Routing-модель сама загрузит нужный skill по совпадению описаний задачи. Прямые ссылки `See also: other-skill` — антипаттерн (та же проблема, что у rule→skill: дублирование description, хрупкость к рефакторингу).
- **Не дублировать rules:** если конвенция уже в rule, не повторять её в skill

#### Чеклист качества skill
Каждый skill должен отвечать на 6 вопросов:
1. **Trigger:** Когда агент должен загрузить этот skill?
2. **Inputs:** Какая информация нужна перед стартом?
3. **Steps:** Пошаговая процедура выполнения
4. **Checks:** Как проверить, что задача выполнена?
5. **Stop conditions:** Когда остановиться и спросить человека?
6. **Recovery:** Что делать, если проверка не прошла?

## Паттерны организации Skills

Skills могут быть организованы двумя способами. Выбор паттерна зависит от объёма и структуры темы.

### Паттерн 1: Цельный skill (один файл)

**Когда использовать:**
- Skill цельный или не слишком большой (<500 строк)
- Все части темы нужны вместе
- Тема не делится на независимые подтемы

**Пример:** [`skills/c35-postgresql/SKILL.md`](../skills/c35-postgresql/SKILL.md) — всё про PostgreSQL в одном файле.

**Преимущества:**
- Простота: всё в одном месте
- Меньше накладных расходов на навигацию
- Агент видит всю картину сразу

### Паттерн 2: Разбитый skill (разводящий + тематические файлы)

**Когда использовать:**
- Много контента (>500 строк в одном файле)
- Контент хорошо бьётся по независимым подтемам
- Подтемы используются в разных сценариях (не всегда нужны вместе)

**Пример:** `grpc/` — основной SKILL.md с навигацией + тематические файлы:
- `grpc/SKILL.md` — обзор, когда использовать, навигация по подтемам
- `grpc/auth.md` — аутентификация (загружается только при работе с auth)
- `grpc/streaming.md` — streaming (загружается только при работе со streaming)
- `grpc/reflection.md` — reflection (загружается только при работе с reflection)

**Преимущества:**
- Экономия контекста: загружается только нужная подтема
- Не сбивает модель лишней информацией
- Прогрессивное раскрытие: от общего к частному

**Концепция:** можно рассматривать как «вложенный skill» — основной SKILL.md играет роль «skill о skills», раскрывающего подтемы по необходимости.

### Критерий выбора

```
Размер < 500 строк И тема цельная
    → Паттерн 1 (один файл)

Размер > 500 строк ИЛИ тема делится на независимые части
    → Паттерн 2 (разбиение)
```

## Требования к дополнительным файлам

Если skill содержит большие артефакты (длинные примеры, шаблоны, справочные таблицы), они выносятся в отдельные файлы внутри директории skill.

### Когда НЕ нужны дополнительные файлы

Если весь skill умещается в SKILL.md (<500 строк) и не содержит больших артефактов — дополнительные файлы не нужны. **Простая структура предпочтительнее.**

### Структура директории
```
skills/
  <skill-name>/
    SKILL.md              # Основной файл (обязательный)
    examples/             # Примеры кода (опционально)
      handler_example.cpp
      test_example.py
    templates/            # Шаблоны файлов (опционально)
      migration.sql
    reference/            # Справочные материалы (опционально)
      config_schema.yaml
```

### Правила для доп. файлов
- **SKILL.md** — единственный обязательный файл. Должен быть самодостаточным для базового сценария
- **Доп. файлы** загружаются агентом по ссылке из SKILL.md, только когда нужны
- **Ссылки** из SKILL.md на доп. файлы — относительные: `See [example](examples/handler_example.cpp)`
- **Размер доп. файла:** без ограничений, но предпочтительно <200 строк
- **Не дублировать:** если файл уже есть в репозитории (шаблон, схема), ссылаться на оригинал, не копировать

### Примеры кода: плейсхолдеры vs встроенный код

**Требование:** примеры в skills должны быть актуальными и собираться (желательно — проверяться в автосборке).

**Проблема прямых ссылок на файлы репозитория:** каждое обращение LLM к внешнему файлу = дополнительный поход агента, лишние токены и время.

**Решение: плейсхолдеры + сборка skill-артефакта**

1. В SKILL.md используем плейсхолдеры вида:
   ```markdown
   [remote example](/path/to/service/handler.cpp:123-145)
   ```
2. При сборке через aisuite:
   - Парсим плейсхолдеры
   - Выдираем код из указанных диапазонов строк
   - Вставляем текстом в финальный skill-артефакт

3. Источник примеров — **эталонный сервис**:
   - Создаём эталонный сервис в нашей кодовой базе
   - Примеры для skills живут в нём
   - Эталон не меняется без ведома команды, поэтому плейсхолдеры остаются валидными

**Плюсы:**
- ✅ Проверяемость сборкой (примеры реально компилируются/тестируются)
- ✅ Актуальность (примеры не устаревают вместе с API)
- ✅ Нет лишних походов LLM в runtime (код уже встроен в skill)

**Минусы:**
- ⚠️ Усложнение поддержки (при смещении строк плейсхолдеры становятся невалидными)
- ⚠️ Требует инфраструктуры для подстановки при сборке

**Компромисс:** эталонный сервис минимизирует риск невалидных ссылок, а линтер при сборке отловит оставшиеся.

## Прогрессивное раскрытие

Архитектура доставки знаний агенту работает в три уровня:

```
Уровень 1 — Discovery (всегда):
  Rule descriptions + Skill descriptions
  ~X токенов на skills, ~Y токенов на rules
  Агент видит ЧТО доступно

      ↓ агент решает что релевантно

Уровень 2 — Activation (по релевантности):
  Полное тело SKILL.md
  50–500 строк, 200–2000 токенов
  Агент знает КАК делать

      ↓ агент решает что нужно подробнее

Уровень 3 — Deep dive (по запросу):
  Доп. файлы из skills/<name>/examples/, reference/
  Без ограничений по размеру
  Агент видит ДЕТАЛИ
```

## Опциональная мета-инструкция про загрузку skills

> **По умолчанию такая секция в rules НЕ нужна** — правила загрузки skills зашиты в базовый промпт агента, и дублировать их в always-on rule означает тратить токены впустую.
>
> Эта секция — **страховка на случай**, когда в конкретном агенте/окружении обнаруживается некорректное поведение (например, перебор skills, загрузка «на всякий случай», игнорирование description). Тогда в один из always-apply rules можно добавить явный дисциплинирующий блок.

Это **не** навигация «rule → конкретный skill» (это антипаттерн, см. «Связь rule и skill»), а общая инструкция про дисциплину discovery:

```markdown
# Навигация по Skills

Skills содержат пошаговые инструкции для конкретных задач.

Загружай skill ТОЛЬКО когда:
- Задача пользователя явно попадает в описание skill
- Тебе не хватает информации для выполнения задачи

НЕ загружай skill когда:
- Ты уже знаешь ответ из текущего контекста
- Пользователь задаёт общий вопрос, не требующий процедуры
- Ты просто хочешь «на всякий случай» посмотреть документацию
```

Добавляем только при подтверждённой проблеме с поведением модели. Убираем, когда проблема исчезает.

## Автоматизация процедур

Если пошаговая процедура в skill:
- Полностью автоматизируема (нет решений человека)
- Выполняется за один вызов
- Имеет чёткие входы/выходы

→ **Рекомендация:** по возможности заверните её в скрипт или MCP tool и опишите использование в skill.

**Пример:**
- ❌ Skill с 6 шагами: «1) cd в директорию, 2) запустить `ya project update`, 3) проверить вывод, 4) если ошибка — сделать X, 5) запустить ya style . 6) закомитить файлы, измененные update и style»
- ✅ Скрипт + skill: MCP tool `run_codegen(service_path)` + короткий skill «Когда и как использовать codegen, как интерпретировать ошибки»

**Преимущества:**
- Меньше ошибок: автоматизация надёжнее ручных шагов
- Экономия токенов: вызов tool вместо длинной инструкции
- Переиспользование: один tool вызывается из разных skills и режимов

**Когда НЕ стоит автоматизировать:**
- В процедуре есть развилки, требующие суждения человека/LLM
- Шаги зависят от контекста задачи, который нельзя формализовать
- Процедура используется редко — стоимость разработки скрипта > выгода

## Анти-паттерны

| Название | Симптомы | Решение |
|----------|----------|---------|
| **Энциклопедия** | Skill >500 строк, выглядит как вики | Разбить на SKILL.md + доп. файлы |
| **Всё-в-одном** | Skill применяется к >50% задач | Перенести в Rule |
| **Тайный skill** | Агент никогда не загружает skill | Переписать description с реальными триггерными словами |
| **Дубликат** | Одна и та же информация в rule И в skill | Выбрать одно место, из другого удалить |
| **Rule-туториал** | Rule содержит пошаговую процедуру на 5+ шагов | Вынести процедуру в skill (или скрипт/MCP tool), в rule оставить суть |
| **Rule-указатель** | Rule содержит прямое упоминание имени конкретного skill/rule | Убрать упоминание, усилить триггеры в description скилла |
| **Хрупкий skill** | Ломается при изменении структуры репо | Использовать переменные, не хардкодить пути |

## Сводная таблица

| | Rule | Skill |
|---|---|---|
| **Когда загружается** | Всегда | По релевантности задачи |
| **Кто триггерит** | Система | Агент (по description) |
| **Стиль** | Императивный: MUST, NEVER | Процедурный: шаг 1, шаг 2... |
| **Размер файла** | ≤200 строк | ≤500 строк (основной) |
| **Суммарный бюджет rules** | ≤X  токенов на все rules | ≤Y  токенов на все секции skills descriptions + Без лимита  на тела (платим при активации) |
| **Примеры кода** | Короткие, 5–10 строк | Развёрнутые, с контекстом при необходимости |
| **Доп. файлы** | Нет; rule самодостаточен | Да, в поддиректориях skill |
